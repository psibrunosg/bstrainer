# Prescribed/performed blocks live in sibling tables per kind, not one polymorphic table

ADR-0001 made `WorkoutTemplate`/`WorkoutSession` hold an ordered list of mixed blocks (exercise, activity, circuit) at the TypeScript layer. At the Postgres layer this could be one polymorphic `workout_blocks` table (single `kind` + `position`, with per-kind detail tables joined 1:1), or sibling tables per kind (`prescribed_exercises`, `prescribed_activities`, `prescribed_circuits`, each with their own `position`), matching how `prescribed_exercises`/`performed_exercises` already work in this schema. We chose sibling tables: it reuses the existing table shape and RLS-policy pattern (join chain up to `training_plans`/`workout_sessions`) instead of introducing a new polymorphic-parent concept, at the cost of `position` not being uniquely enforced *across kinds* within one workout — two blocks of different kinds could claim the same position at the DB level. The application is the source of truth for the merged, sorted block order (it already assigns sequential `order` across the whole mixed list in `instantiateTemplate`); a future polymorphic migration is only worth it if cross-kind position collisions become a real data-integrity problem in practice.
